Testing - Test Criteria: Please select all the check boxes applicable to your team.

Traveller

Team does not take part in scope identification

User Stories have only one acceptance criteria that is defined upfront.

Team applies same logic/technique to all testing activities.

Artisan

Acceptance criteria is only used to derive acceptance tests.

Certain user stories may have more than one acceptance criteria (i.e happy and unhappy path).

Team investigates different test approaches per user story for different scenarios.

A proper Definition of Done is defined for testing tasks. (E.g a Testing Checklist)

No definition of readiness.

Expert

Whole team reviews and defines acceptance criteria for user stories.

Acceptance criteria specifies both functional and non-functional requirements of increment.

Test conditions are extracted from each Acceptance Criterion to help define testing requirements.

Team applies a Risk level to each Test Condition extracted.

Team applies some form of Context Driven approach to testing.

Proper definition of readiness is defined.

Professional

Acceptance Criteria are INVEST(Independent, Negotiable, Valuable, Estimable, Small and Testable).

Test conditions are prioritised based on the risk level.

Testing is focused on the intent behind Software/Application Usage.

Manual testing (such as Exploratory Testing) is conducted based on logical reasoning, heuristics and critical thinking.

Definition of Done and Readiness are reviewed as part of retrospective session.

Cross Feature Team Dependencies addressed in Scrum of Scrums Meeting.

Master

Acceptance criterion have clear Pass/Fail results - No Partial Acceptance.

Test process applies dynamic and progressive risk analysis and is constantly evolving as the team evolves.
